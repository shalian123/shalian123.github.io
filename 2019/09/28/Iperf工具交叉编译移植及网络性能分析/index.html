Iperf工具交叉编译移植及网络性能分析

发表于 2019-09-28   |   分类于 linux   |  

一、案例简述

　　在开发带有WiFi功能的产品时，往往需要对WiFi的性能要特别关注。最近在本组产品的集成测试中发现：当设备连接WiFi时，设备出现监视失败或呼叫室内机时，室内机没画面、web刷新或配置失败等问题的概率会大大增加，而且这个缺陷还是偶现的。为了排查是取流还是网络的问题，我们首先与ClentDemo监视时进行了对比，发现ClentDemo能够正常的监视和取流。因此我们需要进一步确认是否与网络问题有关。常用的ping IP的方式虽然可以查看出网络的延时，但是具体网络情况还是无法直观的得出。常见的一个测试方法是使用iperf工具。

二、案例分析和解决过程

2.1、工具的介绍

　　Iperf 是一个网络性能测试工具。可以测试 TCP 和 UDP 带宽质量，可以测量最大 TCP 带宽，具有多种参数和 UDP 特性，可以报告带宽，延迟抖动和数据包丢失。这里的网络不但可以是一般的以太网，也可以是无线网络。输出结果是网络传输速度，单位是bps。 Iperf 在 linux 和 windows 平台均有二进制版本供自由使用。在同一局域网没有网络透传的设备之间，iperf工具可以测试PC机，室内机，门铃机三者之间任意两者的网络速度。

2.2、iperf移植(交叉编译)

（1）下载源码：https://iperf.fr/iperf-download.php#windows

（2）解压tar zxvf iperf-2.0.5-source.tar.gz
（3）cd iperf-2.0.5目录下，执行 “./configure –host=arm-hisiv500-linux”，生成Makefile文件。
（4）在iperf-2.0.5目录下，运行“make”,将会在iperf-2.0.5/src/下生成iperf程序。（arm平台的）
（5）NFS挂载，将iperf程序拷贝到设备里，或直接打包进去。
　　注意：
　　① arm-hisiv500-linux为H10平台的交叉编译器，具体设备具体对待。
　　② 当执行make时，会报错，如下图：

　　解决方法：
　　将iperf-2.0.5目录下config.h中的“define malloc rpl_malloc”注释掉，重新编译。
　　③ window版的iperf生成方式有两种：一种是执行配置文件时不需要加交叉编译器，即直接运行 “./configure”;另一种是直接到官网下载https://iperf.fr/iperf-download.php#source，如下图：

　　下载完成后，将解压出来的脚本及应用程序复制到%systemroot%目录即可。注意：这个 %systemroot% 目录，可以直接在cmd中，输入echo %systemroot%，即可得到目录，我这里是c盘下的Windows。如果下载的是iperf3，输入命令时也要是iperf3加参数。

2.3、iperf的使用

　　iperf基于C/S模式，在服务器端运行“./iperf -s”；在客户端。运行“./iperf –c 服务端的IP –i 1 –t 10”,参数的意义如下：
　　（1）、-s 以服务器模式启动；
　　（2）、-c 以客户端模式启动，这时需要指定服务器所在的IP地址；
　　（3）、-t 指定测试时间；
　　（4）、-i 指定测试时间间隔，打印结果，例如上例就是以1秒间隔不断输出结果。
　　（5）、-d 双向传输测试；
　　注意：
　　①、确认设备和你的电脑在同一个网段，然后ping通。
　　②、其他参数可以查看iperf自带的帮助文档（iperf -h）。
　　为了测试WiFi的性能，我分别设置了三组对比实验，PC与门口机之间有线和无线连接状态下的网络状况，室内机与门口机之间有线和无线连接状态下的网络状况，还有监视黑屏和监视正常状态下的网络情况。

三、案例总结

　　相同时刻，设备分别连接无线和有线时，网络相差比较悬殊，同时我们连接无线网络时出现监视失败，呼叫黑屏及web刷新或配置失败等问题的概率会大大增加。因此有理由怀疑这些问题的出现和网络性能有关。而由1080P、720P、4CIF、CIF所需要的理论带宽和720p视频需要带宽的计算所知，要想能够正常预览1080p的画面，需要满足4M带宽，而在对比试验中，连接WiFi时，带宽并没有达到4M，且波动较大。
　　同时，此缺陷的偶现现象可能与网络波动的影响有关。与室内机组沟通后了解到黑屏情况与有些关键帧没收到，或复合流（视频、音频）插入顺序紊乱有关。因此可以当此缺陷复现时，可录像分析，进一步研究是否因为网络原因导致关键帧丢失。
